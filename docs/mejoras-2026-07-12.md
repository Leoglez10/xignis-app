# Mejoras 2026-07-12 — Plan a producto profesional

Auditoría completa de la app + alcance nuevo de la reunión de RRHH. Este
documento reemplaza a `mejoras-2026-07-07.md` como backlog vivo y define
**5 fases** (P0 → P4) con tickets priorizados, ramas por fase y Definition of
Done.

---

## Contexto

Se escaneó toda la app: 13 migraciones SQL, 2 edge functions, 66 archivos de
features, 13 archivos de `lib/`, 5 componentes UI, config de Vite/Tailwind/
Capacitor/TS, y 16 docs. El resultado: app sólida (realtime ref-counted,
Supabase tipado, RLS completa, notificaciones por trigger, a11y baseline
fuerte, 20 tests verde), pero le faltan piezas clave para nivel
profesional y para cumplir lo que pidió RRHH.

**Decisiones de la reunión que cambian el alcance:**
1. Plataforma: **web responsiva** (no nativa). Capacitor queda como envoltorio
   opcional pero la prioridad es que corra en navegador desktop y móvil. El
   layout actual tiene gaps en desktop real (sin sidebar de manager, top tabs
   se amontonan, bottom sheets en desktop no escalan a modal central).
2. RRHH debe poder **crear y gestionar áreas/departamentos** dentro de la app.
3. Reports deben poder **filtrarse por área/departamento** (Diseño, Edición,
   etc.) para ver dinámica por área.
4. Tipos de permiso deben distinguir **con goce vs sin goce de sueldo**.
5. Cuando un permiso es aprobado o rechazado, **se debe notificar al empleado
   Y a RRHH**. Hoy el trigger solo notifica al empleado en `approved`/`rejected`
   final; RRHH no se entera del cierre.
6. Perfiles de empleado gestionados por RRHH con **alta/baja y razón de
   separación** (hoy solo hay delete duro sin razón ni soft-delete).
7. Desarrollo iterativo basado en feedback de Leo.

**Otros hallazgos de la auditoría (106 tickets):** bugs de deep link, CSV que
exporta todo y no lo filtrado, reglas en `localStorage` no backend, sin Error
Boundary, sin React Query (todo fetch reimventado), UI kit incompleto, sin
Toast, multi-h1, sin dark mode, sin tests de componentes, sin observabilidad,
sin búsqueda real. Detallados por fase abajo.

```
main

```

Cada rama sale limpia de `main`. PR individual, título `[FASE Px] ...`.

## Fase P0 — Estabiliza (rama `mejora/p0-estabiliza`)

**Requisito reunión:** cuando un permiso se aprueba/rechaza, notificar al

  también recorra `hr_admin` + `admin` e inserte notificación

- **Archivos:** nueva migración, `database.types.ts` (NotificationType),

## Fase P1 — Departamentos y tipos de permiso (rama `mejora/p1-departamentos`)

Alcance nuevo de la reunión: áreas CRUD, filtro por área en reports,
permisos con/sin goce.

### P1-1 Tabla `departments`
- **Migración:** `20260712130000_departments.sql`
  ```sql
  create table public.departments (
    id uuid pk default gen_random_uuid(),
    name text not null unique,
    description text,
    created_at timestamptz default now(),
    updated_at timestamptz default now()
  ```

- **Relación:** agregar `profiles.department_id uuid → departments(id)` nullable.
  Mantener `job_title` como puesto (no confundir con área).
- **RLS:** SELECT para autenticados; INSERT/UPDATE/DELETE solo hr_admin/admin.
- **Realtime:** añadir `departments` a `supabase_realtime`.
- **Seed:** poblar con áreas actuales de la agencia (Diseño, Edición, etc.)
  cuando Leo pase la lista.

### P1-2 Pantalla CRUD de áreas
- **Ruta nueva:** `/admin/departments` → `DepartmentsScreen.tsx`.
- **Funciones:** lista, crear, renombrar, archivar (soft delete con

- **Servicio:** `departmentService.ts` con `listDepartments()`,
  `createDepartment()`, `updateDepartment()`, `archiveDepartment()`,

- **UI:** tabla desktop + cards mobile, igual patrón que `EmployeesScreen`.

### P1-3 Asignar departamento a empleados
- **Cambios:** `EmployeesScreen` `EditSheet` agrega `DepartmentSelect`
  (nullable). `InviteSheet` también. `profiles.department_id` en insert/update.
- **`admin-create-user`** edge function: aceptar `department_id` opcional.
- **`profileService`**: `listEmployees()` hace join `departments` para mostrar
  nombre de área.

### P1-4 Filtro por área en Reports
- **Cambios:** `AdminReportsScreen` agrega `<select>` de áreas (Departamento:
  todos / Diseño / Edición / ...). Filtra `filtered` por
  `employee.department_id`.
- **KPIs por área:** el resumen "resúmenes de permisos por departamento" que
  pidió RRHH → nuevo widget `DepartmentBreakdown` con barras horizontales
  por área (counts de aprobadas/rechazadas/pendientes).
- **Servicio:** `dashboardService.getDepartmentBreakdown()` → agrupa por

### P1-5 Tipos de permiso con/sin goce
- **Requisito reunión:** "Incluir una opción para permisos sin pago junto con
  las opciones de permiso con pago."

  **Decisión de diseño:** dos opciones:
  - (A) Ampliar `leave_type` enum con `vacation_paid`, `vacation_unpaid`,
    `personal_paid`, `personal_unpaid`, `sick_paid`, `sick_unpaid`, `other`.
  - (B) Agregar columna `paid boolean default true` en `leave_requests` y
    mantener `leave_type` como está.

  **Recomendado (B):** menos migración de enum, más flexible. `paid` es
  ortogonal al tipo. El wizard pregunta tipo Y ticking "con goce/sin goce".

- **Migración:** `20260712140000_leave_requests_paid.sql`
  ```sql
  alter table public.leave_requests
    add column paid boolean not null default true;
  ```
- **UI wizard:** `LeaveRequestScreen` `StepType` agrega toggle "Con goce de
  sueldo" (default on) después del radio de tipo.
- **DB types:** `LeaveRequest.paid: boolean`, `LeaveRequestDraft.paid`.
- **Reports:** KPI "días con goce vs sin goce", filtro por `paid`.
- **Balance:** `getMyVacationBalance` cuenta solo `paid=true` vacation days.

### P1-6 Seed de departamentos
- **Acción:** cuando Leo pase la lista, agregar a `seed_departments.sql`
  inserts. Mientras tanto, areas vacías + el CRUD permite crear.

### Definition of Done P1
- RRHH crea/edita/archiva áreas desde la app.
- Empleados asignados a áreas.
- Reports filtran y agrupan por área.
- Wizard pregunta con/sin goce.

- Realtime actualiza lista de áreas.

## Fase P2 — Responsive desktop (rama `mejora/p2-responsive-desktop`)

La app hoy es mobile-first con algo de responsive pero hay gaps serios en
desktop real. La reunión confirmó que la prioridad es web responsiva en PC.

### P2-1 Layout system por rol
- **Empleado:** ya es mobile-first; en desktop centrar contenido en
  `max-w-xl` (ya hecho en algunas screens). Verificar consistencia.
- **Manager:** hoy no tiene sidebar. En desktop (`lg:`) agregar
  `ManagerSidebar` con nav (Inicio, Equipo, Agenda, Perfil) + contenido en
  `max-w-5xl`. Mobile mantiene top tabs.
- **Admin/RH:** ya tiene `AdminSidebar` pero solo se usa en algunas screens.
  Unificar todas las screens admin bajo `AdminShell` (ya existe). Verificar
  `AdminRulesScreen`, `AdminReportsScreen` usen el shell.

### P2-2 Sidebar manager component
- **Nuevo:** `src/features/manager/components/ManagerSidebar.tsx` y
  `ManagerShell.tsx` (paralelo a `AdminShell`).
- **Nav items:** Inicio (`/manager`), Equipo (`/manager/team`), Agenda
  (`/manager/calendar`), Perfil (`/profile`).
- **Mobile:** `ManagerBottomNav` se oculta al scroll (mismo patrón que
  `AdminBottomNav` con `useScrollDirection`).
- **`App.tsx`:** rutas `/manager/*` envueltas en `<RequireAuth allowedRoles=
  ["manager"]>` que ya renderiza `<TopBar/>` — ajustar para que en desktop
  TopBar ceda paso al sidebar.

### P2-3 `BottomSheet` → `Modal` en desktop
- **Bug:** `BottomSheet` cuenta como bottom drawer siempre. En desktop un
  modal centrado es más correcto.
- **Fix:** `BottomSheet` ya tiene `sm:mb-4 sm:max-w-md sm:rounded-[28px]`
  que lo centra en desktop. Verificar que no sea `fixed bottom-0` en sm+.
  Mejorar: en `lg:` usar `Modal` centrado con overlay, no drawer.
- **Patrón:** `BottomSheet` acepta prop `variant: "drawer" | "modal"`, auto
  por breakpoint.

### P2-4 Tablas desktop reales
- **`AdminReportsScreen`** hoy oculta tabla en mobile y muestra cards. En
  desktop la tabla existe pero es rawHTML. Componentizar `DataTable` con
  columnas configurables + sort + sticky header.
- **`EmployeesScreen`** en desktop mostrar tabla (hoy siempre cards).
  Toggle cards/table por breakpoint.

### P2-5 TopBar no se amontona en desktop
- **Bug:** TopBar tabs en mobile `overflow-x-auto` sin indicador. En desktop
  caben todos pero no hay separación clara contextual.
- **Fix:** en desktop los tabs van al sidebar, TopBar en desktop solo
  muestra avatar + título + notif + switcher. En mobile mantiene tabs
  inferiores (o el bottom nav del shell).

### P2-6 `navConfig.titleForPath` mejora
- **Bug:** detalle routes (`/manager/member/:id`) retornan "Inicio" falso.
- **Fix:** mapeo de títulos por pattern o que cada screen setee su título
  via context. Propuesta: `usePageTitle("Detalle de Miembro")` hook que
  actualiza el h1 del TopBar.

### P2-7 Layout breakpoints audit
- Revisar todas las `section` container classes: `min-h-dvh`, `px-5`, `pb-28`
  son mobile. En desktop deben ser `max-w-*xl mx-auto` + padding balanceado.
  Documentar tokens: `--content-max-employee`, `--content-max-manager`,
  `--content-max-admin`.

### P2-8 PWA meta + installable
- **`index.html`** agregar `manifest.webmanifest`, `apple-touch-icon`,
  `apple-mobile-web-app-capable`, `apple-mobile-web-app-status-bar-style`.
- Unificar `theme-color` con Capacitor backgroundColor.
- Hace la web instalable en desktop Chrome y mobile Safari.

### Definition of Done P2
- Manager ve sidebar en desktop ≥ `lg` (1024px).
- Bottom sheets mutan a modal centrado en desktop.
- Tablas admin visibles y usables en desktop.
- Títulos de detalle correctos.
- Web instalable como PWA.
- Lighthouse responsive 100.

## Fase P3 — Alta / Baja con razón de separación (rama `mejora/p3-alta-baja`)

Requisito reunión: "perfiles de empleados gestionados exclusivamente por RRHH
que soporten registros de alta/baja con las razones de separación."

### P3-1 Soft-delete + `employment_status`
- **Migración:** `20260712150000_employment_status.sql`
  ```sql
  alter table public.profiles
    add column employment_status text not null default 'active'
    check (employment_status in ('active','on_leave','terminated','archived')),
    add column terminated_at timestamptz,
    add column termination_reason text,
    add column separation_type text
    check (separation_type is null or separation_type in
      ('voluntary','involuntary','end_contract','relocation','retirement','other'));
  ```
- **Razones:** `termination_reason` texto libre (o enum amplio). Reunión no
  cerró el set — empezar con enum flexible.

### P3-2 Pantalla de baja
- **Ruta:** dentro de `EmployeesScreen` `DeleteEmployeeSheet` evoluciona a
  `SeparationSheet` con wizard de 3 pasos:
  1. Aviso (hoy existe).
  2. Confirmar nombre + ack history (hoy existe) **+** seleccionar
     `separation_type` (voluntary/involuntary/end_contract/...) + texto
     `termination_reason`.
  3. Ejecutar: `admin-delete-user` edge function actualizada para
     **soft-delete** (set `employment_status='terminated'`, `terminated_at`,
     `termination_reason`) en vez de `auth.admin.deleteUser` duro.

- **Decisión de producto:** ¿soft-delete (mantener auth.users + datos
  históricos) o hard-delete (limpiar todo)? Reunión dice "registros de
  alta/baja con razones" → **soft-delete** recomendado. Auth user se
  deshabilita via `admin.auth.admin.updateUserById(id, { ban_duration: 24
  })` (ban) y profile archivado.

### P3-3 Pantalla de alta
- **Hoy:** `InviteSheet` (admin-create-user) ya hace alta. Extendido en P1
  con `department_id`.
- **Mejora:** `InviteSheet` muestra `separation_type` historial si el email
  ya existió (reactivar). Revisar si `upsert` en profiles revive un perfil
  archivado.

### P3-4 Filtros de estado de empleo
- **`EmployeesScreen`:** filter chips "Activos / En permiso / Terminados /
  Archivados / Todos". Default "Activos".
- **Reports:** filtro por `employment_status`.
- **Dashboard admin:** widget "Bajas recientes (30d)" paralelo a
  `RecentOnboardingsWidget`.

### P3-5 Historial laboral
- **Tabla nueva (futuro):** `employment_events (id, user_id, event_type,
  effective_date, reason, metadata jsonb, created_by)`. Registra alta,
  cambios de departamento/manager/baja.
- **P3:** crear tabla + trigger que inserta en `employment_events` cuando
  `employment_status`, `department_id`, `manager_id` cambian.
- **Pantalla:** `ProfileScreen` admin muestra timeline de eventos laborales.

### P3-6 Datos para futuras expansiones
- **Reunión:** "historial salarial, fechas de entrada, información de
  seguridad social en actualizaciones futuras."
- **`profiles.hire_date`** ya existe (migración 11). Usarlo para
  `employment_status='active'` start.
- **Placeholder columnas nullable:** `profiles.social_security_number text`,
  `profiles.salary_history jsonb` (no llenar todavía, solo schema). Decidir
  si se agregan en P3 o待 futura fase.

### Definition of Done P3
- RRHH da de baja con razón de separación.
- Empleado terminado no puede loguearse (auth ban).
- Datos históricos conservados (soft-delete).
- Filtros por estado de empleo en employees + reports.
- Widget "bajas recientes" en dashboard admin.
- Historial laboral visible en profile admin.

  invalidadas por realtime channel. Borrar `mountedRef` patterns.
- **Reduce:** ~300 líneas de código, mejora UX offline, agrega cache.
- **Archivos:** todos los screens de features, nuevo `src/lib/queryClient.ts`.

### P4-2 UI kit completo
Componentes nuevos en `src/components/ui/`:

  `ManagerSelect`/`DepartmentSelect`).

- `Avatar` (subir `EmployeeAvatar`, centralizar `initials(name)` con edge
  cases tipo "de la Fuente").
- `Badge`/`Chip` (status pills con tone prop).
- `Card` base.

### P4-3 Eliminar duplicación de fuentes de verdad

- Hover literals (`bg-slate-100`, `bg-emerald-300`) → tokens.

### P4-4 Toast + confirm global

### P4-7 Settings screen

### P4-8 Wizard draft persistente

## Reglas de producto actualizadas (docs/rules.md futuro)

- `leave_type` se complementa con `paid` boolean. El wizard pide ambos.
- `profiles.department_id` asigna área. RRHH gestiona departamentos.
- `employment_status` distingue active/on_leave/terminated/archived.
- Baja requiere `separation_type` + `termination_reason`.
- RRHH recibe notificación en cada cierre de solicitud (approve/reject).
- Web responsiva: desktop ≥ 1024px usa sidebar, mobile usa tabs/bottom nav.
- No nativo-only featuresBlocking web build.

## Cómo ejecutar

Cuando des luz verde:
1. `git checkout -b mejora/p0-estabiliza main`
2. Implementar P0 completa (commits atómicos por ticket `P0-N descripción`)
3. `npm run build && npm test` verde
4. PR #1 a main, pedir review de Leo
5. Iterar feedback
6. Merge → siguiente fase

Cada fase es independiente y mergeable por separado. Si una fase genera
conflictos con la anterior, rebase de main.
