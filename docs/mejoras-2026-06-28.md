# Mejoras 2026-06-28 — Front, notificaciones, paneles y flujos

Documenta el trabajo de esta iteración: revisión de pantallas mobile/desktop,
animaciones iOS, sistema de notificaciones, flujos de auth completos, alta de
usuarios por RH, panel de jefe con equipo, tests y endurecimiento de la base.

La app corre dentro de Capacitor (webview iOS/Android) sobre React 18 + Vite +
TypeScript. El enfoque visual prioriza iOS.

## 1. Sistema de animaciones (suaves y elegantes)

Definido en `src/styles/globals.css`. Curvas de aceleración estilo iOS:

- `--ease-ios: cubic-bezier(0.32, 0.72, 0, 1)`
- `--ease-soft: cubic-bezier(0.22, 1, 0.36, 1)`

Keyframes: `fade-up`, `fade-in`, `scale-in`, `slide-up`, `backdrop-in`.

Clases utilitarias:

| Clase | Uso |
|-------|-----|
| `.animate-fade-up` | Entrada de headers, tarjetas y secciones (460 ms) |
| `.animate-fade-in` | Aparición suave de overlays/contenido |
| `.animate-scale-in` | Paneles y modales (BottomSheet) |
| `.animate-slide-up` | Hojas inferiores |
| `.animate-backdrop` | Fondo oscuro de los modales |
| `.stagger > *` | Lista con retardo incremental (40–390 ms, hasta 8 hijos) |
| `.press` | Feedback táctil: `scale(0.97)` en `:active` |

**Accesibilidad:** todo se neutraliza bajo `@media (prefers-reduced-motion: reduce)`.

Pantallas con animaciones aplicadas: dashboard de empleado, dashboard de jefe,
dashboard de RH, directorio de empleados, equipo del jefe, y todos los modales
(BottomSheet con scale-in + backdrop).

## 2. Notificaciones (en producto + realtime)

### Base de datos (`supabase/migrations/20260628232814_notifications_and_routing.sql`)

- Tabla `public.notifications` con RLS: el usuario solo lee/actualiza las suyas.
  **No hay policy de INSERT para clientes** — solo los triggers `SECURITY DEFINER`
  escriben notificaciones.
- Índice `notifications_user_created_idx (user_id, created_at desc)`.
- Publicación realtime: `alter publication supabase_realtime add table public.notifications`.

### Triggers que generan notificaciones

- `notify_new_leave_request` (AFTER INSERT): avisa al jefe (o a RH si no hay jefe).
- `notify_leave_status_change` (AFTER UPDATE OF status): avisa a RH al aprobar el
  jefe, y al empleado en cada cambio (aprobada / rechazada / pasó a RH).

### Front

- `src/features/notifications/services/notificationService.ts`:
  `listNotifications`, `markNotificationRead`, `markAllNotificationsRead`,
  `subscribeToNotifications(userId, onInsert)` (canal realtime por usuario).
- `src/features/notifications/NotificationBell.tsx`: campana con badge de no leídas,
  panel en BottomSheet, marcado optimista, tiempo relativo con `date-fns` (es).
  Integrada en los tres dashboards (empleado, jefe, RH).

## 3. Ruteo automático de solicitudes

`route_leave_request` (BEFORE INSERT): si el empleado **no tiene jefe asignado**,
la solicitud salta directo a `pending_hr` en vez de `pending_manager`.

Flujo de estados completo:

```
pending_manager ─┬─> approved_by_manager ─> (RH) ─┬─> approved
                 │                                 └─> rejected
                 └─> rejected_by_manager
(sin jefe) ─────────> pending_hr ────────────────> approved / rejected
```

## 9. Endurecimiento de la base

`supabase/migrations/20260628233000_harden_trigger_functions.sql`: se revoca
`EXECUTE` a `anon` y `authenticated` sobre las funciones-trigger
(`route_leave_request`, `notify_new_leave_request`, `notify_leave_status_change`)
para que no sean invocables vía RPC de PostgREST.

### Advertencias preexistentes (no de esta iteración)

Quedan avisos del linter sobre el esquema legacy en español (`empleados`,
`login_intentos`, `solicitudes_*`, `tokens_aprobacion`) con RLS sin policies, la
función `update_updated_at` con search_path mutable, el bucket público
`xignis-profiles` y la protección de contraseñas filtradas deshabilitada en Auth.
El esquema en español **no lo usa la app** (la app usa `profiles` /
`leave_requests` / `leave_request_approvals`). Recomendado: limpiar el esquema
legacy o añadir policies, y activar "Leaked password protection" en Auth.

## 11. Perfil propio + edición con restricciones

`src/features/profiles/screens/ProfileScreen.tsx` (`/profile`): pantalla de perfil
para **cualquier rol**. Muestra avatar (iniciales), nombre, correo (solo lectura,
vive en `auth.users`), rol y puesto. El usuario edita su **nombre**; cerrar sesión
vive aquí.

Acceso: en el dashboard de **empleado** y de **jefe** se toca el avatar/ícono de
perfil del header; en **RH** está en la barra de navegación (sidebar y barra
inferior).

### Restricciones (seguridad en la base, no solo en UI)

`supabase/migrations/20260628234000_profiles_self_update_guarded.sql`:

- Policy `profiles_self_update`: el usuario puede actualizar su propia fila.
- Trigger `guard_profile_privileged_fields` (BEFORE UPDATE): si el rol **no** es
  `admin`/`hr_admin`, fuerza `role`, `manager_id` y `job_title` a sus valores
  anteriores. Así un empleado no puede auto-promoverse ni reasignarse jefe aunque
  llame al UPDATE directo. RH/admin sí pueden cambiar esos campos (vía
  `EmployeesScreen` → `updateProfileAssignment`).

Servicio: `updateMyProfile({ full_name?, avatar_url? })` y `getCurrentEmail()` en
`profileService.ts`.
